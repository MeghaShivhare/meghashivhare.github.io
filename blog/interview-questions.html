Kubernetes & EKS 19 questions
What are the components of the Kubernetes control plane?Medium▶
  • API Server (kube-apiserver) — the front door. All kubectl commands, internal components, and external tools talk to it. Validates and persists state to etcd.
  • etcd — distributed key-value store that holds all cluster state (pods, services, configmaps, secrets). The single source of truth.
  • Scheduler (kube-scheduler) — watches for unscheduled pods and assigns them to nodes based on resource requirements, affinity rules, and taints/tolerations.
  • Controller Manager (kube-controller-manager) — runs controller loops: Deployment controller, ReplicaSet controller, Node controller, etc. Continuously reconciles desired vs actual state.
  • Cloud Controller Manager — AWS-specific controllers: provisions load balancers, manages EBS volumes, handles node lifecycle with EC2.
EKS-specific: AWS runs the control plane in a separate managed VPC. You don't have direct access to etcd or the API server nodes — AWS manages HA, patching, and backups.
Explain etcd in detail.Hard▶
  • What it is: Distributed, consistent key-value store. Stores all Kubernetes cluster state — every pod, service, configmap, secret, deployment, etc.
  • Consensus: Uses the Raft consensus algorithm. One leader handles all writes; followers replicate. Quorum requires (N/2)+1 members — always use an odd number of nodes (1, 3, 5).
  • Watch mechanism: Clients subscribe to changes on a key prefix. This is how K8s controllers work — they watch etcd for changes and react. Every kubectl get --watch is an etcd watch.
  • resourceVersion: Every object in etcd has a resourceVersion. Used for optimistic concurrency — if two writers modify the same object, the second write is rejected.
  • Backup/restore: etcdctl snapshot save backup.db and etcdctl snapshot restore backup.db. Critical to back up before cluster upgrades.
In EKS: You cannot access etcd directly. AWS manages it. Back up by exporting critical manifests to Git (GitOps) rather than relying on etcd snapshots.
How do you upgrade an EKS cluster version without downtime?Hard▶
  1. Check release notes — review K8s changelog for deprecated APIs and breaking changes. Test your manifests against the new version.
  2. Upgrade control plane first — via EKS console or aws eks update-cluster-version. AWS handles this with minimal disruption; API server may be briefly unavailable but pods keep running.
  3. Update add-ons — kube-proxy, CoreDNS, VPC CNI must be updated to versions compatible with the new K8s version. Do this after control plane upgrade.
  4. Update node groups (rolling) — managed node groups support rolling updates: new nodes are provisioned with the new AMI/version, then old nodes are cordoned and drained one by one. PodDisruptionBudgets ensure replicas stay available during drain.
  5. Verify — check all pods are Running, all add-ons healthy, kubectl get nodes shows new version.
Zero-downtime key: Set PodDisruptionBudgets on all critical workloads before upgrading. Without a PDB, drain can evict all replicas of a deployment simultaneously.
What is the difference between desired state and current state in Kubernetes?Easy▶
  • Desired state: what you declare in your YAML manifest — "I want 3 replicas of this pod running image:v2."
  • Current state: what is actually running in the cluster right now.
  • Reconciliation loop: K8s controllers continuously compare desired vs current state and take actions to close the gap. If a pod crashes, the ReplicaSet controller sees current=2, desired=3 and creates a new pod.

This is the core of Kubernetes: declarative, self-healing. You declare the end state, K8s figures out how to achieve it and keeps it that way.

How data is managed in StatefulSets

  • Each pod gets a unique PersistentVolumeClaim (PVC) — e.g. data-postgres-0, data-postgres-1.
  • When a pod restarts (even on a different node), Kubernetes reattaches the same PVC — the data survives.
  • PVCs are NOT deleted when the StatefulSet is deleted — must be cleaned up manually to avoid data loss.

Cross-question: using a database while storing state

For managed databases (RDS, Aurora), pods connect via a service endpoint — the state is external to the cluster entirely. The pod is stateless; the database is not in Kubernetes. For in-cluster databases (PostgreSQL StatefulSet), each pod mounts its own EBS-backed PVC via a StorageClass.

Node Affinity, Node Selector, Pod Affinity, and Pod Anti-AffinityMedium▶
  • NodeSelector: simplest form. Key-value label match on nodes. Binary — either the node has the label or it doesn't. disktype: ssd
  • Node Affinity: more expressive version of NodeSelector. Two types: requiredDuringSchedulingIgnoredDuringExecution (hard — pod won't schedule if not satisfied) and preferredDuringScheduling… (soft — scheduler tries but continues if not possible). Supports operators: In, NotIn, Exists, Gt.
  • Pod Affinity: schedule this pod near other pods that match a label selector (same node or same zone). Use case: app pod colocated with its Redis sidecar for low latency.
  • Pod Anti-Affinity: spread pods away from each other. Use case: ensure two replicas of a deployment never land on the same node — improves HA.
Cordon, Drain, Uncordon, and NoExecute taint — explain eachMedium▶
  • Cordon: kubectl cordon <node> — marks the node as Unschedulable. No new pods will land on it. Existing pods keep running undisturbed.
  • Drain: kubectl drain <node> --ignore-daemonsets --delete-emptydir-data — gracefully evicts all pods (respecting PodDisruptionBudgets and terminationGracePeriodSeconds). Automatically cordons the node first.
  • Uncordon: kubectl uncordon <node> — makes the node schedulable again after maintenance.
  • NoExecute taint: stronger than cordon. Evicts pods that don't have a matching toleration, either immediately or after tolerationSeconds. Applied automatically when a node goes NotReady — pods without the toleration are evicted after 300 seconds by default.
Maintenance sequence: cordon → drain → perform maintenance → uncordon.
Labels and Selectors in KubernetesEasy▶
  • Labels: key-value pairs attached to any K8s object. app: frontend, env: prod, version: v2. Used to organize and select objects.
  • Selectors: queries that filter objects by their labels. A Service uses a selector to find the pods it routes traffic to. A Deployment uses a selector to manage its ReplicaSet pods.
  • Equality-based: app=frontend — exact match.
  • Set-based: env in (prod, staging) — more expressive, used in affinity rules and node selectors.

Labels are the glue of Kubernetes — Services, Deployments, NetworkPolicies, and HPA all use label selectors to target the right set of pods.

Service discovery failing across namespaces — no DNS issues. Troubleshoot.Hard▶

Root cause: When reaching a service in another namespace, you must use the Fully Qualified Domain Name (FQDN), not just the service name.

  • Service exists: kubectl get svc -n <target-namespace> — verify the service name and port.
  • Endpoints: kubectl get endpoints -n <target-namespace> — if empty, pods aren't matching the service selector.
What are Init Containers?Easy▶

Containers that run to completion before the main application container starts. They run sequentially — init-1 must complete before init-2 starts, which must complete before the main container starts.

Use cases: wait for a database to be ready, run database migrations, download config from S3, set file permissions that the main container needs.

If an init container fails, the pod restarts and tries again. The main container never starts until all init containers succeed.

What is a DaemonSet and when do you use it?Easy▶

A DaemonSet ensures exactly one pod runs on every node (or every node matching a nodeSelector). When nodes are added, the pod is automatically added. When nodes are removed, the pod is garbage collected.

Use cases: log collectors (Fluentd, Filebeat), node monitoring agents (Node Exporter, Datadog agent), networking plugins (CNI), security agents.

Drain behavior: DaemonSet pods are NOT evicted during kubectl drain by default. Use --ignore-daemonsets flag when draining to proceed anyway.
Sidecar pattern — what is it and how does log shipping work?Medium▶

A sidecar is a second container in the same pod that augments the main container. They share the same network namespace (localhost) and can share storage volumes.

Log shipping pattern: main container writes logs to a shared emptyDir volume → sidecar container tails that volume and ships logs to a log aggregator (CloudWatch, Loki, Elasticsearch). The main app doesn't need to know about the logging destination.

Other sidecar uses: Istio/Envoy service mesh proxy, config reloader, metrics scraper.

What is the Kubeconfig file?Easy▶

The file (~/.kube/config) that kubectl reads to know how to connect to a cluster. Contains three types of entries:

  • clusters: API server URL + CA certificate
  • users: credentials (client cert, token, or exec plugin like aws-iam-authenticator)
  • contexts: a named binding of cluster + user + optional namespace
  • current-context: which context kubectl uses by default
common kubeconfig commands
kubectl config get-contexts          # list all contexts
kubectl config use-context my-cluster # switch context
kubectl config current-context       # show active context
aws eks update-kubeconfig --name my-cluster --region eu-north-1  # add EKS cluster
Does Prometheus run as a DaemonSet or a regular Pod?Medium▶

It depends on which Prometheus component — this is a trick question:

ComponentControllerWhy
Prometheus ServerDeployment / StatefulSetSingle central collector/storage. HA via Thanos.
Node ExporterDaemonSetMust run on every node to collect that node's CPU/memory/disk metrics
kube-state-metricsDeploymentQueries K8s API, doesn't need to be per-node
AlertmanagerDeployment / StatefulSetHA pair for deduplication and routing
The answer interviewers are testing: Node Exporter is the DaemonSet. Prometheus server itself is a Deployment/StatefulSet running as a regular pod.
How does pod-to-pod and pod-to-node communication work?Medium▶
  • Pod-to-pod (same node): via a virtual ethernet pair (veth) through the node's network bridge. Pods on the same node communicate directly over the bridge — no NAT, no external hop.
  • Pod-to-pod (different nodes): via the CNI plugin (aws-node / VPC CNI in EKS). Each pod gets a real VPC IP. Traffic between pods on different nodes goes through the VPC network directly.
  • Pod-to-itself (localhost): containers within the same pod share a network namespace. They communicate via localhost:port. This is how sidecar containers communicate with the main container.
  • Pod-to-service: via kube-proxy, which maintains iptables/ipvs rules. The service's ClusterIP is a virtual IP — kube-proxy rewrites traffic to a real pod IP based on the service's endpoint list.

Stateful vs Stateless creation sequence in KubernetesMedium▶

Deployment (stateless): pods are created in parallel, in any order, with random name suffixes. Scale-down kills pods randomly.

StatefulSet (stateful): pods are created sequentially — pod-0 must be Running and Ready before pod-1 starts. Scale-down terminates in reverse order (pod-2 before pod-1 before pod-0). This ordering matters for databases where pod-0 is typically the primary and must exist before replicas join.

Docker & Containers 10 questions
Explain a Dockerfile and its key instructionsEasy▶

A text file with step-by-step instructions to build a Docker image. Each instruction creates a layer in the image.

Dockerfile
FROM node:18-alpine          # base image
WORKDIR /app                 # set working directory
COPY package*.json ./        # copy dependency files
RUN npm ci --only=production # install deps (creates a layer)
COPY . .                     # copy source code
ENV NODE_ENV=production      # set env var
EXPOSE 3000                  # document the port (doesn't publish)
USER node                    # run as non-root for security
CMD ["node", "server.js"]    # default start command
What is a multi-stage Dockerfile and why use it?Medium▶

Uses multiple FROM statements. Only the final stage ends up in the image — all build tools, compilers, and test dependencies are discarded.

multi-stage Dockerfile
FROM node:18 AS builder          # build stage — has npm, node_modules
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build                # produces /app/dist

FROM nginx:alpine                # final stage — tiny, no node
COPY --from=builder /app/dist /usr/share/nginx/html
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

Result: final image is ~25MB (nginx:alpine + built assets) instead of ~400MB (full node image with all dependencies).

How do you minimize Docker image size?Medium▶
  1. Use slim/alpine base images — node:18-alpine (50MB) vs node:18 (1GB).
  2. Multi-stage builds — build tools never enter the final image.
  3. Combine RUN commands — each RUN creates a layer. Chain commands with && and clean up in the same layer:
efficient RUN
RUN apt-get update \
    && apt-get install -y curl \
    && rm -rf /var/lib/apt/lists/*   # clean in same layer!
  1. .dockerignore — exclude node_modules/, .git/, test files, local env files.
  2. Don't install debug tools in production images — no curl, wget, bash unless needed.
  3. Specific COPY — copy only what the app needs, not the whole repo root.

What does the CMD instruction do in Docker?Easy▶

Defines the default command that runs when a container starts. Can be overridden at docker run time.

  • Shell form: CMD npm start — runs as /bin/sh -c "npm start". Process doesn't receive Unix signals (SIGTERM) properly.
  • Exec form (preferred): CMD ["node", "server.js"] — process runs directly, receives signals, enables graceful shutdown.

CMD vs ENTRYPOINT: ENTRYPOINT defines the fixed executable; CMD provides default arguments to it. If both are set, CMD args are appended to ENTRYPOINT.

What is Docker Compose and when do you use it?Easy▶

Tool for defining and running multi-container applications via a docker-compose.yml file. One command (docker compose up) starts your whole stack — app + database + cache + any other services.

Use for: local development and testing. Not for production — use Kubernetes for production orchestration.

Services communicate using their service names as DNS hostnames (Docker manages an internal DNS). Volumes are shared between services.

Docker cp — copy files between container and hostEasy▶

Copy files or directories between a running (or stopped) container and the host filesystem.

docker cp
docker cp container_name:/app/logs/error.log ./local-error.log   # container → host
docker cp ./config.yaml container_name:/app/config/config.yaml   # host → container
Docker cp doesn't reflect real-time changes. For live config updates, prefer environment variables, ConfigMaps (in K8s), or volume mounts rather than copying files in at runtime.

AWS Services 17 questions
EC2 is in a private subnet — how do you make it publicly accessible?Medium▶

Best practice: don't expose the EC2 directly. Put a load balancer in public subnets in front of it:

  • Deploy an ALB in public subnets (subnets with an IGW route).
  • Create a target group pointing to the private EC2.
  • ALB listener forwards inbound traffic to the EC2 — the EC2 stays private.

For SSH/admin access: use AWS Systems Manager Session Manager — no public IP needed, no bastion host, full CloudTrail audit log.

If you must expose directly: move to a public subnet and assign an Elastic IP, or add an IGW route + public IP. Not recommended for production.

EC2 needs to fetch data from S3 — how? (VPC endpoint + IAM role)Medium▶

Two recommended approaches — use both together for best security and cost:

  • IAM Role / Instance Profile: attach an IAM role to the EC2 with S3 read permissions. The EC2 automatically gets temporary credentials via the instance metadata service (IMDS). No credentials stored on disk. Best practice.
  • VPC Gateway Endpoint for S3: add a gateway endpoint to your VPC route table. Traffic to S3 stays within the AWS network — doesn't go through NAT Gateway or internet. Free to use. Eliminates NAT Gateway data charges for S3 traffic.
Never do this: store AWS_ACCESS_KEY_ID / AWS_SECRET_ACCESS_KEY on the EC2 instance. If the instance is compromised, those credentials are stolen and can't be automatically rotated.
What is an Instance Profile?Medium▶

A container for an IAM role that can be attached to an EC2 instance. When you "attach an IAM role to EC2" in the console, you're technically attaching an Instance Profile that wraps that role.

  • Credentials are automatically rotated by the metadata service every few hours.
  • The EC2 can call http://169.254.169.254/latest/meta-data/iam/security-credentials/<role-name> to get current credentials (AWS SDKs do this automatically).
  • In Terraform: create aws_iam_role → aws_iam_instance_profile → attach to aws_instance via iam_instance_profile.

Cross-zone load balancing — what is it?Easy▶

Without cross-zone: each load balancer node only distributes traffic to targets in its own AZ. If AZ-A has 2 targets and AZ-B has 8 targets, AZ-A targets get 4× more traffic each.

With cross-zone: each LB node distributes traffic evenly across all targets in all AZs — true even distribution regardless of AZ target count.

  • ALB: enabled by default, no extra charge.
  • NLB: disabled by default. Enabling incurs cross-AZ data transfer charges.
S3 — Legal Hold, Retention, Bucket Policy, ACLs, Canonical IDHard▶

Object Lock (S3)

  • Retention Period: prevents object deletion or overwrite for a fixed period. Can be extended but never shortened (even by root). Two modes: Governance (admin with special permission can bypass), Compliance (nobody can override — not even root).
  • Legal Hold: indefinite lock with no expiry date. Toggle on/off independently of retention. Removed by someone with s3:PutObjectLegalHold permission.

Access control

  • Bucket Policy: resource-based JSON policy on the bucket. Can allow access from other AWS accounts, services, or based on conditions. Most flexible — AWS recommends using this.
  • ACLs: older per-object or bucket-level permissions. AWS now recommends disabling ACLs and using bucket policies instead. Object Ownership setting controls this.
  • Canonical ID: a long unique alphanumeric string identifying an AWS account. Used in cross-account ACL grants when you don't want to expose account IDs or email addresses.

CloudWatch Composite Alarms — what are they?Medium▶

Combine multiple CloudWatch alarms using boolean logic (AND, OR, NOT) to create a single alarm state.

Example: fire only when BOTH HighCPU AND HighLatency are in ALARM state — reduces false positives from CPU spikes that don't actually affect users.

Use case: suppress alert noise. Instead of getting paged for every individual metric spike, only alert when multiple symptoms occur together — indicating a real incident vs a transient blip.

CloudFormation Drift vs Change SetsMedium▶
  • Drift Detection: someone manually changed a resource outside CloudFormation (via console or CLI). CFT detects this as drift. Stack → Detect Drift → shows which resources differ from the CFT template and how. Fix by either importing the change into the template or reverting the manual change.
  • Change Set: a preview of what will change before you execute a stack update. Like terraform plan. Shows resources that will be Added / Modified / Removed. Execute the change set to apply. Never update a production stack without reviewing a change set first.
Assume Role (STS) — what is it and when do you use it?Medium▶

sts:AssumeRole allows a principal (IAM user, service, another role, EC2, Lambda) to temporarily take on another IAM role and receive short-lived credentials with that role's permissions.

Use cases:

  • Cross-account access: your account assumes a role in another account to deploy resources or read data.
  • CI/CD: GitHub Actions OIDC assumes a deployment role — no long-lived access keys stored in GitHub.
  • EKS IRSA: pod service account assumes an IAM role to access AWS services without EC2 instance credentials.
  • Least privilege: human users assume elevated roles only when needed, for limited duration (1-12 hours).

Amazon Glue, Kinesis, CloudFront, AWS Config — brief overviewMedium▶
  • Amazon Glue: serverless ETL service. Crawlers scan data sources (S3, RDS) and build a Data Catalog (schema). ETL jobs (Python/Spark) transform raw data for analytics. Used in data pipelines feeding Athena or Redshift.
  • Kinesis Data Streams: real-time data streaming. Producers push records (logs, events, metrics); consumers (Lambda, EC2) read them in order. Retention 1–365 days. Use when you need real-time, ordered processing.
  • Kinesis Firehose: fully managed delivery pipeline — stream → S3, Redshift, OpenSearch. No consumer code needed. Buffers and batches automatically.
  • CloudFront: CDN. Caches content at edge locations globally. Origins: S3, ALB, EC2, API Gateway. Path-based behaviors, WAF integration, OAC for private S3. ACM certs for CloudFront must be in us-east-1.
  • AWS Config: records configuration changes to AWS resources over time. Evaluate resources against compliance rules. Useful for auditing "who changed this security group and when." Not real-time monitoring — use CloudTrail + CloudWatch for that.
Networking & DNS 6 questions
How does DNS routing work? Walk me through a request.Medium▶
  1. User types example.com in browser.
  2. Browser checks its local DNS cache. If found and TTL not expired → done.
  3. OS checks /etc/hosts, then system resolver cache.
  4. Query goes to the Recursive Resolver (configured DNS server — ISP, 8.8.8.8, etc.).
  5. Resolver doesn't know example.com. Asks a Root Nameserver (.) → "who handles .com?"
  6. Root nameserver returns the address of the .com TLD nameserver.
  7. Resolver asks the TLD nameserver → "who's authoritative for example.com?"
  8. TLD returns the authoritative nameserver for example.com.
  9. Resolver asks the authoritative nameserver → gets the A record (IP address).
  10. Resolver returns IP to client, caches it for the TTL duration. Browser connects to the IP.
VPC Traffic Flow — how does traffic move through a VPC?Medium▶
  • Internet → Public subnet EC2: Internet → IGW → Route table (0.0.0.0/0 → igw) → Subnet NACL → Security Group → EC2.
  • Private subnet EC2 → Internet (outbound): EC2 → Route table (0.0.0.0/0 → nat-gw) → NAT Gateway (in public subnet) → IGW → Internet.
  • EC2 to EC2 (same VPC): via the VPC local route (10.0.0.0/16 → local). Stays within the VPC network, no external hops.
  • EC2 to S3 (with VPC endpoint): via the gateway endpoint — route table entry pl-xxx (S3 prefix list) → vpce-xxx. Never leaves AWS network.

Each hop evaluates: Route Table → NACL (subnet-level, stateless) → Security Group (instance-level, stateful).

Ingress and Target Groups in Kubernetes/AWSMedium▶

Kubernetes Ingress: an API object that defines HTTP/HTTPS routing rules into cluster services. Rules: host + path → service. Requires an Ingress Controller (nginx, AWS ALB Ingress Controller) to actually implement the rules as a real load balancer.

AWS Target Group: a set of registered targets (EC2 instances, IP addresses, Lambda functions, ECS tasks) that an ALB routes traffic to. Each target group has its own health check settings. ALB listener rules forward to target groups.

How they connect in EKS: an Ingress resource → AWS Load Balancer Controller creates an ALB → registers pods as IP targets in a target group. Pod IP is the target, not the node IP (in IP target mode).

VPC default resources — what does AWS create automatically?Easy▶

When you create any VPC:

  • Default Security Group — allows all traffic from within the same SG, denies all inbound from outside.
  • Default NACL — allows ALL inbound and outbound traffic (rule 100 allow *).
  • Default Route Table — contains the local route for the VPC CIDR.

The default VPC (pre-created per account per region) also has: an Internet Gateway attached, a public subnet in each AZ with auto-assign public IP enabled, and a route table entry 0.0.0.0/0 → igw.

Hybrid connectivity — how do you connect on-premises to AWS?Medium▶
  • VPN (Site-to-Site): encrypted IPsec tunnel over the public internet. Quick to set up (~hours). Bandwidth limited (~1.25 Gbps). Variable latency. Good for lower-bandwidth or quick connectivity needs.
  • Direct Connect: dedicated private fiber connection from on-premises to AWS. 1–100 Gbps. Consistent, low latency. Takes weeks to provision. Best for high-bandwidth, latency-sensitive workloads.
  • Transit Gateway: hub to connect multiple VPCs and on-premises networks. VPN or Direct Connect attaches to TGW, then all VPCs connected to TGW can reach on-premises.

How do you split weighted traffic in AWS?Medium▶

  • ALB weighted target groups: an ALB listener rule can forward to multiple target groups with specified weights. E.g. 90% to v1 target group, 10% to v2 target group.
  • Argo Rollouts (K8s): progressive canary delivery controller. Automatically shifts traffic percentage, runs analysis jobs (error rate, latency), auto-promotes or rolls back.
  • Istio/service mesh: VirtualService with weighted routes between service versions within the cluster.

Key metrics for service reliability — what do you monitor and prioritize?Medium▶

The Four Golden Signals (Google SRE):

  • Latency: time to serve a request. Measure p50, p95, p99 — never just the average. One slow request hides in the average.
  • Traffic: requests per second, transactions per second — how much demand is on the system.
  • Errors: rate of failed requests (5xx responses, failed transactions). The most user-visible signal.
  • Saturation: how full is the system? CPU, memory, queue depth, disk. Saturation predicts future degradation.

Prioritization: alert on symptoms (high error rate, high latency) not causes (high CPU). A CPU spike that doesn't affect users isn't an emergency. If error rate is fine, the high CPU can wait for business hours.

SLI, SLO, SLA — what are they and why do they matter?Medium▶
  • SLI (Service Level Indicator): a specific metric measuring service behavior. Example: "percentage of requests completed in under 200ms over the last 30 days."
  • SLO (Service Level Objective): the target for an SLI. Example: "99.9% of requests must complete in under 200ms." Your internal engineering target.
  • SLA (Service Level Agreement): a contract with customers with financial consequences if violated. Usually looser than SLO — your internal target should be stricter than what you promise customers.

Error budget = 1 - SLO. At 99.9% SLO: 0.1% budget = ~43 minutes downtime/month. When the error budget is nearly exhausted, freeze new deployments and focus on reliability. Error budget protects reliability without blocking all progress.

How would you handle a latency spike on a critical application?Hard▶
  1. Triage scope: is it all requests or specific endpoints? All users or a region? Check load balancer access logs for which paths are slow.
  2. Check recent changes: any deployments in the last hour? Terraform applies? Config changes? Roll back if correlated.
  3. Resource bottleneck: CPU, memory, network I/O on application pods and database. kubectl top pods, CloudWatch metrics.
  4. Database: most latency spikes trace back to slow queries, lock contention, or connection pool exhaustion. Check DB slow query log, active connections, wait events.
  5. Downstream dependencies: check if a downstream service is slow or returning errors. Distributed tracing (X-Ray) shows where time is spent.
  6. Scaling: if traffic spiked, scale out pods (HPA) or add read replicas to relieve DB load.
  7. Communicate: update incident channel with findings every 15-20 minutes. Don't go dark while investigating.
How do you conduct a post-incident review (post-mortem)?Medium▶

Key principle: blameless. The goal is systemic improvement, not finding someone to blame.

  • Timeline: reconstruct a minute-by-minute timeline of what happened, when it was detected, and when it was resolved.
  • Impact: quantify — how many users affected, what revenue/operations impact, how long.
  • Root cause: use "5 Whys" to go beyond the immediate cause to the underlying systemic issue. Not "the pod crashed" — why did it crash? Why didn't monitoring catch it earlier?
  • Contributing factors: what conditions made the incident possible or made it worse?
  • Action items: specific, owned, time-bound improvements. "Add DB connection pool monitoring by Friday (Owner: @name)." Not vague items like "improve monitoring."

AWS Backup — what is it and when do you use it?Easy▶

Centralized, policy-driven backup service for AWS resources. Instead of configuring backup schedules individually per service, AWS Backup lets you define a single backup plan that covers multiple resource types.

Supported resources: EBS volumes, RDS databases, DynamoDB tables, EFS file systems, S3 buckets, EC2 instances, Aurora, DocumentDB, Neptune, FSx, Storage Gateway.

Key features: backup plans with CRON schedules, retention policies, cross-region and cross-account copy for DR, point-in-time restore, compliance reporting via AWS Backup Audit Manager.

Use when: you need consistent backup policy across multiple AWS services and accounts, or need to demonstrate compliance (SOC2, HIPAA) with backup controls.

Terraform variable precedence — .tf file, tfvars, JSON, CLI flagsHard▶

From lowest priority to highest priority (later sources override earlier ones):

  1. Default values in variable blocks in .tf files lowest
  2. terraform.tfvars file (auto-loaded)
  3. terraform.tfvars.json file (auto-loaded)
  4. *.auto.tfvars and *.auto.tfvars.json files (alphabetical order)
  5. TF_VAR_* environment variables
  6. -var-file=<file> flag on CLI
  7. -var="key=value" flag on CLI highest
Interview answer: "CLI flags always win. tfvars files are loaded automatically. The variable block default is only used if nothing else sets the value."
Cloud architecting — reliability strategiesSystem Design▶
  • Multi-AZ: spread resources across 2+ AZs. Survive AZ failure with automatic failover (RDS Multi-AZ, NLB, EKS managed node groups).
  • Multi-region: for disaster recovery or latency. Active-active (both regions serve traffic) or active-passive (failover to second region).
  • Immutable infrastructure: never patch running servers — replace them. Bake AMIs/container images, roll out new ones, terminate old ones.
  • Infrastructure as Code: all resources defined in Terraform/CFT. Enables consistent recreation, audit trail, and peer review of changes.
  • Circuit breakers: stop calling a failing service. Let it recover instead of cascading failures. Tools: Istio, Resilience4j.
  • Health checks everywhere: ALB target health, K8s readiness/liveness probes, RDS enhanced monitoring.
  • Backup and recovery testing: automated backups aren't enough — regularly test restore procedures.
What would you do first in a pod/server crash — rollback or fix?Hard▶

Rollback first, in almost every case.

  • The primary objective in an incident is to restore service — minimize MTTR (Mean Time to Recovery).
  • A broken system while you're still debugging extends the outage for users.
  • Rollback is fast and predictable. Finding the root cause takes unknown time.
  • You can safely debug the rolled-back version after the incident is over.

When to fix instead of rollback:

  • The fix is a one-line change and faster than a rollback.
  • Rollback would cause data loss (e.g. DB migration already ran and can't be reversed).
  • No stable previous version exists to roll back to.
  • The issue isn't deployment-related (infra failure, dependency outage).
Standard sequence: rollback → restore service → write incident notes → root cause analysis → proper fix → staged deploy with monitoring.

SSL/TLS certificates — how do they work?Medium▶
  • What TLS does: encrypts data in transit between client and server. Prevents eavesdropping and man-in-the-middle attacks.
  • Certificate contains: server's public key, domain name, issuing CA, validity period, and the CA's digital signature.
  • Chain of trust: certificate signed by an intermediate CA → signed by a root CA. Browsers trust root CAs built into the OS/browser. If the chain is valid, the cert is trusted.
  • TLS handshake (simplified): client connects → server presents cert → client verifies cert chain → they negotiate cipher and exchange session keys → encrypted communication begins.
  • In AWS: ACM (Certificate Manager) issues free managed TLS certs. Handles renewal automatically. CloudFront and ALB integrate directly with ACM. ACM certs for CloudFront must be in us-east-1.
  • Self-signed: cert signed by itself, not a trusted CA. Browsers show a warning. Acceptable for internal services where you control both sides.

Nice value — what is it, what command, inversely proportional?Medium▶

The nice value controls a process's CPU scheduling priority. Range: -20 (highest priority) to +19 (lowest priority). Default for new processes: 0.

Inversely proportional to priority: a lower (more negative) nice value = higher CPU priority. A process with nice -10 gets more CPU time than one with nice +10.

nice commands
nice -n 10 my-command           # start a process with nice value 10 (lower priority)
nice -n -5 my-command           # start with nice value -5 (higher priority, needs root)
renice -n 5 -p 1234             # change nice value of running process (PID 1234)
renice -n -10 -p 1234           # only root can set negative nice values
